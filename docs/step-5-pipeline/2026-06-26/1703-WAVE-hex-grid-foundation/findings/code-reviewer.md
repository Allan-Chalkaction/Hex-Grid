VERDICT: APPROVE

23 files, 6 tickets reviewed; build + lint + secret-scan clean, RLS migration correct across all three documented traps; 4 low/nit findings, none blocking.

## Findings
- **CR-001** [low · none · DEFER] src/lib/tenant.ts exports getActiveTenantId() and listMemberships() but neither is imported by any component (git grep confirms zero consumers in src/ outside tenant.ts's own definition + a doc comment in auth.ts). AC-006's verification clause says 'signing in as the dev user yields a resolved active tenant_id in the app' — but the app never resolves an active tenant: SiteList relies entirely on RLS auto-scoping (supabase.from('site').select), and AuthGate only tracks session, not tenant context. AC-007's literal wire-to-consumer requirement is satisfied (the resolver module exists and reads membership; onAuthStateChange/getSession are consumed by AuthGate), and the substantive tenant-isolation guarantee holds in the database via RLS, so this is not a correctness bug for W1's empty-read-path ACs. But tenant.ts is currently dead code — it ships untested by any runtime path and the 'resolved active tenant_id in the app' phrasing of AC-006 is only met in the database, not in the app layer. Why it matters: dead exports drift out of sync with the schema and the AC-006 intent ('working tenant context on first run') is only proven at the DB level, not exercised by the UI. **Target ticket:** W2 (site CRUD wave, which is the first consumer of active tenant_id). **Summary:** wire getActiveTenantId() into the app's tenant-context (e.g. AuthGate resolving and providing tenant_id) when W2 introduces a write path that needs it, or drop listMemberships() if it stays unused.
- **CR-002** [low · none · DISMISS] supabase/seed.sql inserts directly into auth.users and auth.identities with a hand-rolled column set. This is inherently coupled to GoTrue's internal schema and can break on a CLI/GoTrue upgrade (e.g. newer GoTrue adds an `email` generated column on auth.identities, or additional NOT-NULL columns on auth.users). Why it matters: a future `supabase` CLI bump could make `supabase db reset` fail on the seed, breaking AC-001. Mitigating factors that make this acceptable for W1: (a) the author explicitly documented the GoTrue NULL-token scan issue and set token columns to '' deliberately; (b) the README ships Path B (operator creates the dev user via Studio/CLI first) as the production-shaped, schema-stable fallback, so a broken self-contained seed has a documented workaround; (c) it is gated to local-dev only and idempotent. **Dismissal rationale:** the fragility is real but consciously chosen and documented, with a first-class fallback path (Path B) that does not touch auth internals. This is the standard local-dev tradeoff for a self-contained `db reset`; flagging it as a change-request would override a documented, defensible implementer decision. No rule violation.
- **CR-003** [nit · none · DISMISS] supabase/config.toml sets site_url = 'http://127.0.0.1:3000' and additional_redirect_urls to the :3000 host, but vite.config.ts pins no server port so the dev app runs on Vite's default :5173. The mismatch is inert for W1 because auth.email.enable_confirmations = false and the flow is email/password (no redirect email round-trip), so login is unaffected — but it will bite the moment a magic-link / OAuth redirect is added (the redirect host won't match the running app). **Dismissal rationale:** no functional impact on any W1 AC (confirmations disabled, password flow has no redirect). Cosmetic/forward-looking only; cheaper to fix when a redirect-based flow actually lands. Recommend aligning site_url to the real dev port (or setting `server.port` in vite.config.ts) at that time.
- **CR-004** [nit · none · DISMISS] vite build emits a 1.76 MB (485 kB gzip) main chunk and warns 'Some chunks are larger than 500 kB' — driven by deck.gl + maplibre-gl bundled into the single entry. AC-004 only requires `vite build` to exit 0 (it does) with no console errors, so this does not violate any acceptance atom. **Dismissal rationale:** bundle-size/code-splitting is the performance-reviewer's domain, not code-review scope, and it is expected for a deck.gl+maplibre map shell at W1. Noted for awareness; a dynamic-import split is a reasonable later optimization but out of scope here.
