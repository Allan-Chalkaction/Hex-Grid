## Findings
- **SA-001** [low · none · DEFER] supabase/seed.sql (lines 35-93) inserts a dev auth user directly into auth.users with a hardcoded bcrypt-hashed password (`crypt('devpass123', gen_salt('bf'))`) plus a matching auth.identities row, and binds it to the dev tenant via membership. This is the Path-A self-contained local-dev convenience, gated behind `supabase db reset` (which loads `sql_paths = ["./seed.sql"]` per config.toml line 65) and never runs on a deployed/hosted Supabase project. The credential is explicitly documented as LOCAL DEV ONLY in both the seed header (line 23) and README (line 52). This is NOT a committed production secret and does not violate AC-010 (the password is a known throwaway dev credential, not a service-role/anon key). Risk is purely defense-in-depth: if a future wave reuses seed.sql against a shared/remote environment, or `supabase db push`/`db reset` is ever pointed at a non-local DB, a known-password authenticated user with an `owner` membership would exist. Remediation: keep the seed local-only (it already is — `supabase db reset` is local-stack-scoped), and add a guard comment or a `current_database()`/host assertion that aborts the auth.users insert if run against anything but the local dev DB; alternatively document in README that seed.sql must never be included in a remote migration/seed path. No code change required for W1 correctness; this is a forward guardrail.
- **SA-002** [nit · none · DISMISS] supabase/config.toml line 171 sets `minimum_password_length = 6`, the Supabase default. For a local-dev-only auth seam in W1 this is immaterial (the only account is the documented dev user). Noting only so it is a conscious choice when the parent app's real identity provider is swapped in per the AC-007 seam design — at that point password policy moves to the production identity source and this local-stack value is irrelevant. No action for this wave.
